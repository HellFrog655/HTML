1️⃣
แสดงสินค้า (P1) — ดึงข้อมูลสินค้าจากคลัง แสดงให้ลูกค้าดู

แผนผังที่ 5
ขั้นตอนการทำงานทีละขั้น
Activity Diagrams
การไหลข้อมูล ▼
🍳
เปรียบได้กับสูตรการทำอาหาร — บอกว่าทำขั้นที่ 1 ก่อน แล้วค่อยขั้นที่ 2 ถ้าไม่ผ่านเงื่อนไขให้ย้อนกลับทำใหม่ ถ้าผ่านไปต่อได้
มีทั้งหมด 5 สูตรการทำงาน
①
ลูกค้าส่งฟอร์มขอใบเสนอราคา
เปิดหน้าสินค้า → กรอกชื่อ-เบอร์-บริษัท → กดส่ง → ระบบตรวจสอบข้อมูล → ถ้าครบ: บันทึกและส่งอีเมล Admin → แสดง "ส่งเรียบร้อย" → ถ้าไม่ครบ: แจ้ง error ให้กรอกใหม่
②
ลูกค้าสมัครหลักสูตร
เลือกหลักสูตร → กรอกข้อมูล → ระบบตรวจว่าที่นั่งว่างไหม → ถ้าว่าง: ยืนยัน → ถ้าเต็ม: แจ้งให้รออบรมรอบถัดไป
③
Admin เข้าสู่ระบบ
กรอก username + password → ระบบตรวจสอบ → ถ้าถูก: ได้รับ "บัตรดิจิทัล" (JWT) เข้าแดชบอร์ด → ถ้าผิด: แจ้งเตือนและล็อคหลัง 5 ครั้ง
④
Admin จัดการ Lead
เปิดรายการ Leads → เลือก Lead → โทรหาลูกค้า → กดเปลี่ยนสถานะ "ติดต่อแล้ว" → บันทึก → หรือ ติดตามต่อ → ปิดงาน
⑤
Admin เพิ่มบทความข่าว
กดเพิ่มบทความ → เขียนเนื้อหา → บันทึกเป็นร่าง → ตรวจทาน → กดเผยแพร่ → ขึ้นเว็บทันที
💡
แผนผังนี้เป็น คู่มือการทำงาน ของระบบ ทีมพัฒนาใช้เพื่อให้แน่ใจว่าสร้างครบทุกกรณี รวมถึงกรณีที่ผิดพลาดด้วย
💬

Sequence Diagrams
การไหลข้อมูล ▼
📞
เปรียบได้กับบันทึกการโทรศัพท์ — ใครพูดก่อน พูดว่าอะไร อีกฝ่ายตอบว่าอะไร ทีละประโยคตามลำดับเวลา
มี 3 บทสนทนาหลัก
①
ลูกค้าส่งฟอร์ม — ลูกค้ากดส่ง → หน้าเว็บตรวจสอบ → ส่งไปที่ Server → Server บันทึกลงฐานข้อมูล → ฐานข้อมูลตอบกลับ → Server ตอบหน้าเว็บ → หน้าเว็บแสดง "สำเร็จ" ให้ลูกค้า (7 ขั้นตอน)
②
Admin Login — Admin กรอกรหัส → หน้าเว็บส่งให้ Server → Server ดึงรหัสเข้ารหัสจากฐานข้อมูล → เปรียบเทียบ → ถ้าถูก: ออกบัตรดิจิทัล (JWT) → Admin เข้าแดชบอร์ดได้ (8 ขั้นตอน)
③
สมัครหลักสูตร — ลูกค้าสมัคร → Server ตรวจที่นั่ง → ถ้าว่าง: บันทึก → ส่งอีเมลยืนยัน → ถ้าเต็ม: แจ้งกลับ (6 ขั้นตอน)
💡
ต่างจาก Activity Diagram ตรงที่ เห็นว่าแต่ละฝ่ายรับผิดชอบส่วนไหน — หน้าเว็บทำอะไร, Server ทำอะไร, ฐานข้อมูลทำอะไร แยกกันชัดเจน

สถานะที่เปลี่ยนได้ — เหมือนสัญญาณไฟ
State Diagrams
การไหลข้อมูล ▼
🚦
เปรียบได้กับสัญญาณไฟจราจร — มีได้แค่ "แดง เหลือง เขียว" และเปลี่ยนได้เฉพาะตามเงื่อนไขที่กำหนด จะข้ามจากแดงไปเขียวทันทีไม่ได้
ระบบมี 4 สิ่งที่มีสถานะเปลี่ยนได้
📝
บทความข่าว
"ร่าง" → (กดเผยแพร่) → "เผยแพร่แล้ว" → (กดลบ) → "ลบแล้ว" · หรือ "ร่าง" → "ลบแล้ว" ได้เลย · ลบแล้วกู้คืนไม่ได้
📥
Lead ผู้ติดต่อ
"ใหม่" → (โทรหา) → "ติดต่อแล้ว" → (ติดตาม) → "Follow up" → ปิดได้ "Win" / ปิดไม่ได้ "Lose" · ย้อนกลับจาก Win/Lose ไม่ได้
🎓
การสมัครหลักสูตร
"รอยืนยัน" → (Admin ยืนยัน) → "ยืนยันแล้ว" → (วันอบรมผ่าน) → "เสร็จแล้ว" · หรือ "รอ/ยืนยัน" → (ยกเลิก) → "ยกเลิก"
🤿
ไลฟ์การ์ด
"ลงทะเบียน" → "ว่าง" → (มีงาน) → "ปฏิบัติงาน" → (งานจบ) → "ว่าง" ได้อีก · หรือ → "อบรม" → กลับมา "ว่าง"
💡
แผนผังนี้บอกว่า ข้อมูลแต่ละชิ้นเปลี่ยนสถานะยังไงได้บ้าง และมีเงื่อนไขอะไรถึงจะเปลี่ยนได้ ป้องกันการเปลี่ยนสถานะผิดลำดับ เช่น ลูกค้ายังไม่โทรแต่กดปิดงานแล้ว

Component Architecture

🏠

🏗️

🔒
หน้า Admin (9 หน้า) — Dashboard, จัดการข่าว, Leads, หลักสูตร, ไลฟ์การ์ด, ตั้งค่า ทุกหน้าตรวจสิทธิ์ก่อนแสดง

Functional Requirements (FR)
ข้อกำหนด ▼
📜
เปรียบได้กับTOR (Terms of Reference) หรือสัญญาจ้าง — เขียนชัดเจนว่า "ระบบต้องทำสิ่งนี้ได้" แต่ละข้อตรวจสอบได้ว่าทำเสร็จหรือยัง
ตัวอย่างข้อกำหนด (มีทั้งหมด 23 ข้อ)
FR-001
หน้าแรกต้องแสดงสินค้า 3 รายการ, ข่าว 3 ชิ้น และ Footer ครบ · ความสำคัญ: สูง
FR-003
ฟอร์มขอใบเสนอราคาต้องรับ 6 ข้อมูล และตรวจสอบก่อนส่งทุกครั้ง · ความสำคัญ: สูง
FR-004
เมื่อมี Lead ใหม่ระบบต้องส่งอีเมลแจ้ง Admin ภายใน 60 วินาที · เฟส 2
FR-010
Admin ต้องล็อกอินก่อนเข้าทุกหน้า Admin ถ้ารหัสผิด 5 ครั้งให้ล็อค 15 นาที
FR-015
หน้าเว็บต้องโหลดได้บนมือถือทุกขนาดหน้าจอ ปุ่มกดได้ง่าย ไม่ต้อง zoom
💡
นี่คือ "สัญญา" ระหว่างเจ้าของโปรเจกต์กับทีมพัฒนา ถ้าทำครบทุกข้อถือว่าส่งมอบงานสำเร็จ

Authorization Matrix
ข้อกำหนด ▼
🔑
เปรียบได้กับระบบ key card ในโรงแรม — บัตรพนักงานทั่วไปเข้าได้แค่บางส่วน บัตรผู้จัดการเข้าได้ทุกที่ แผนผังนี้ระบุว่ากุญแจไหนเปิดประตูไหนได้
กฎสำคัญ
✅
ลูกค้าเข้าได้ — ดูสินค้า, ดูบทความที่เผยแพร่แล้ว, ส่งฟอร์มติดต่อ, ดูหลักสูตร
❌
ลูกค้าเข้าไม่ได้ — ดูบทความที่ยังเป็นร่าง, เห็น Leads คนอื่น, แก้ไขข้อมูลใดๆ

⚠️
Rate Limit — ส่งฟอร์มได้ไม่เกิน 3 ครั้ง/นาที/IP ป้องกัน spam อัตโนมัติ
🚫

💡
แผนผังนี้ป้องกันไม่ให้ข้อมูลลูกค้าหลุดไปยังคนที่ไม่มีสิทธิ์ — เป็นพื้นฐานของความปลอดภัยระบบ

API Payload Schemas
ข้อกำหนด ▼
📦
เปรียบได้กับมาตรฐานการส่งพัสดุ — ต้องติดป้ายชื่อผู้รับ ชื่อผู้ส่ง รหัสไปรษณีย์ ห้ามส่งโดยไม่มีที่อยู่ กำหนดว่า "กล่อง" ของข้อมูลต้องมีรูปแบบอะไรบ้าง
ตัวอย่างการส่งฟอร์มติดต่อ
📤
หน้าเว็บต้องส่งไป — ชื่อจริง, นามสกุล, อีเมล (บังคับ), เบอร์ 9-10 หลัก (บังคับ), บริษัท, รุ่นสินค้า, ข้อความ (ไม่บังคับ)
📥
เมื่อสำเร็จ Server ตอบกลับ — "ส่งเรียบร้อย" + รหัส Lead ที่ได้รับ (เก็บไว้ track ได้)
❌
เมื่อข้อมูลผิด Server ตอบกลับ — บอกว่าช่องไหนผิด เช่น "email ไม่ถูกต้อง" แบบระบุตำแหน่ง
💡
ทั้งทีม Frontend และ Backend ต้องตกลง schema นี้ก่อน — ป้องกันปัญหาที่ข้อมูลส่งไปแล้ว Server ไม่รู้จักเพราะชื่อ field ต่างกัน

Traceability Matrix
ข้อกำหนด ▼
🧵
เปรียบได้กับการสอบทานว่า "ทุก requirement มีคนรับผิดชอบแล้ว" — เหมือนตาราง RACI ที่ตรวจว่าทุกงานมีเจ้าของ ไม่มีงานหลุดหาย
ตารางนี้เชื่อมโยงทุกอย่างเข้าหากัน
🔗
FR-003 (ฟอร์มติดต่อ) → Use Case UC-04 → Activity Flow 1 → API: POST /api/contact → ฐานข้อมูล: leads → Component: ContactForm → ทดสอบ: AC-003
🔗
FR-010 (Admin Login) → UC-10 → Activity Flow 3 → API: POST /api/auth/login → ฐานข้อมูล: admins → Component: LoginForm → ทดสอบ: AC-010
ทำไมต้องมีตารางนี้? ถ้า requirement ข้อหนึ่งไม่มี API, ไม่มีฐานข้อมูล, หรือไม่มีการทดสอบรองรับ — แสดงว่างานยังไม่ครบ ตารางนี้ตรวจสอบให้อัตโนมัติว่าไม่มีข้อไหนหลุดหาย
💡
เหมือน ตาราง cross-check ว่าทุก requirement มีคนสร้าง มีที่เก็บข้อมูล และมีวิธีทดสอบครบ — ระบบนี้ผ่าน 100% ครบทุก 23 ข้อ

Glossary
อ้างอิง ▼
📕
เปรียบได้กับศัพท์เฉพาะในสัญญา — ก่อนเซ็น ทนายอธิบายว่า "การยกเลิก" หมายถึงอะไรในสัญญานี้ เพื่อป้องกันการตีความต่างกันภายหลัง
คำศัพท์สำคัญที่ใช้ในโปรเจกต์

🏥 ศัพท์ธุรกิจ Prosperous

  • AED — เครื่องช็อคไฟฟ้าหัวใจ (กดปุ่มเอง)
  • Auto AED — ช็อคอัตโนมัติ ไม่ต้องกดปุ่ม
  • CPR — การปั๊มหัวใจช่วยชีวิต
  • ILSE — มาตรฐานไลฟ์การ์ดสากล
  • Lead — คนที่ส่งฟอร์มสนใจ ยังไม่ซื้อ
  • CSR — กิจกรรมเพื่อสังคมขององค์กร

💻 ศัพท์เทคนิค (แปลแล้ว)

  • API — ช่องทางที่หน้าเว็บคุยกับ Server
  • JWT — บัตรดิจิทัลพิสูจน์ตัวตน Admin
  • Supabase — ฐานข้อมูลออนไลน์
  • Vercel — บริษัท Hosting เว็บ
  • UUID — รหัสประจำตัวที่ไม่ซ้ำกันเลย
  • CRUD — สร้าง/ดู/แก้/ลบข้อมูล
💡
ทุกคนในโปรเจกต์ ตั้งแต่เจ้าของบริษัทถึงนักพัฒนา ต้องใช้คำศัพท์ตรงกัน ป้องกันความเข้าใจผิดที่อาจทำให้สร้างผิดฟีเจอร์
